01

Stack Mismatch — ADR / TS vs. Reality

Resolved
Docs updated to Next.js App Router

Both documents previously opened with the scope line "React + Vite + Tailwind + shadcn/ui + Zustand" and mandated React Router v6 for routing. ADR-00X and TS-00X have since been updated: scope lines now reference Next.js App Router, the routing section has been fully rewritten around file-system segments and next/navigation, and a new BFF section documents the Route Handler pattern for data access.

The docs and the actual stack are now aligned. The remaining routing gap is a code implementation gap, tracked separately below.

02

URL Routing — Not Implemented

High
Docs are now correct — code is not yet implemented

ADR-00X now correctly states: "The browser URL becomes the source of truth for navigation state." The routing section has been fully rewritten around Next.js App Router segments. The doc gap is closed. The code gap is not.

There is exactly one Next.js route — app/page.tsx at /. Surface switching and card expand/collapse exist only in Zustand and localStorage; the URL never changes.

Current — all state in store
/ → everything
surface: localStorage only
expandedId: in-memory only
no deep links · no back button
Target — Next.js App Router
/internal → SummaryFeed
/client → ClientView
/internal/[id] → expanded card
browser history · shareable URLs

The surface store's localStorage persistence is currently doing the job the URL should be doing. Concretely: if you send a colleague a link to a specific opportunity card, they land on the default surface with nothing expanded. The ADR explicitly identifies this as unacceptable.

02b

API / BFF Layer — Not Implemented

High
ADR mandates Route Handlers; none exist for opportunity data

The updated ADR-00X now defines a BFF pattern: client components must access data through Next.js Route Handlers, never by reaching backend services directly. Route Handlers own credentials, data transformation, and the stable UI contract.

The current implementation has no Route Handlers for opportunity data. The only existing Route Handler is app/api/items/route.ts, which serves the original Firestore items collection and is unrelated to the Tortelligence UI. All opportunity data still comes from hardcoded constants imported at module load time — which also means the data layer gap (Section 03) and the BFF gap are the same problem seen from two angles.

Closing both gaps means: create a Route Handler (e.g. app/api/opportunities/route.ts) that reads from Firestore and returns UI-shaped opportunity records; have server components call it on page load, and client components call it for post-hydration fetches. The static OPPORTUNITIES constant in lib/opportunities.ts becomes the seed/fallback only.

03

Mock Data — Where It Comes From

High
All data is hardcoded TypeScript constants

Every opportunity, module definition, and solvency value lives in lib/opportunities.ts as static exports imported directly into useOpportunityView at module load time. No fetch, no server action, no Firestore.

// lib/opportunities.ts — the entire data layer right now
export const OPPORTUNITIES: Opportunity[] = [
  { id: "TG-2588", module: "pharma", label: "Harmful Drug A", score: 91, ... },
  // 11 more static records
]

The existing Firestore infrastructure — lib/firestore.ts, lib/items.ts, the getItems() function — is completely bypassed by the Tortelligence UI. It was the foundation of the original proto-next and is now dead code relative to this feature.

Since OPPORTUNITIES is a static import consumed by a "use client" hook, the full dataset ships in the browser JS bundle. For 12 records this is harmless; for a real portfolio with hundreds of cases it's the wrong architecture.

Three specific data model issues
  • A. SOLVENCY_STATE is a detached parallel map. Every component that needs solvency calls SOLVENCY_STATE[opp.id] ?? "Unknown". It should be opp.solvency — a typed field on Opportunity. The current approach couples two separate data structures by a shared string key with no enforcement.
  • B. ClientView hard-codes the pharma module. Line 10 of client-view.tsx: useOpportunityView("pharma"). The client's subscribed module is a literal in the component, not data-driven. In production this comes from the client record (which law firm subscribed to which module).
  • C. No async loading or error states. The architecture has no provision for async data — no loading skeleton, no error boundary, no empty state for a real fetch that returns zero results.
04

Store Contract — Mostly Compliant

Largely Pass
✓ state / actions split — both stores follow the TS-00X contract exactly. OpportunityStore and SurfaceStore each have a serializable state object and a named actions object.
✓ Feature-scoped, not mega-store — opportunity UI state and surface preference are separated into two purpose-built stores. No single god-object holding unrelated state.
✓ Persistence configured correctly — useSurfaceStore uses persist with name, version: 1, and partialize to store only intentionally durable state. Transient state in useOpportunityStore is correctly left unpersisted.
✓ Feature hooks wrap selectors — useOpportunityView and useSurface select minimal state and expose stable callbacks. Components don't reach into stores directly.
Action naming — generic setters Low

TS-00X says actions must be "domain-named and intent-revealing." setModule and setQuery describe mechanism, not intent. Preferred alternatives: applyModuleFilter(module), clearModuleFilter(), applySearch(query). This makes state traces and code search meaningfully more readable.

Surface store has no migration function Minor

version: 1 is declared but there's no migrate function defined. TS-00X requires a migration strategy when persistence is versioned. For v1 this is low-risk (there's no v0 state to migrate from), but the pattern needs to be in place before a v2 shape change.

05

Stateless Renderer Rule — Compliant

Pass
✓ OppCard owns no state — receives expanded as prop, emits onToggle callback. The expand state lives in useOpportunityStore.state.expandedId, not inside the card.
✓ SummaryFeed and ClientView — both consume useOpportunityView() feature hook with no local useState for business state.
✓ InfoDot tooltip — delegates open/close to Radix UI internally. Tooltip visibility is implementation-detail state and is a named TS-00X exception ("transient focus or hover").
✓ shadcn/ui Input in ClientView — value is wired to external query state, onChange calls setQuery. Correctly externally controlled per the standard.
Shared query state across surfaces Medium

useOpportunityStore.state.query is global. If a user types "Device" in SummaryFeed and switches to ClientView (which is hard-pinned to pharma), the query is still active and silently returns zero results. This isn't a TS violation — the state is correctly externalized — but it's a latent UX bug. Either scope query to the active surface, or clear it on surface switch via a setSurface side-effect.

06

Styling Convention — Meaningful Deviation

Medium
Custom components bypass Tailwind entirely

TS-00X requires that styling "remain aligned with the project Tailwind configuration, tokens, and approved primitives." Every Tortelligence custom component — OppCard, SummaryFeed, AppShell, ClientView, EvidenceLayer, and all primitives — uses inline CSS with CSS variable references instead of Tailwind classes.

// What's in the codebase
style={{ fontFamily: "var(--font-dm-mono)", fontSize: 10, letterSpacing: "1.8px" }}

// What TS-00X expects
className="font-mono text-xs tracking-widest"

The shadcn/ui components (Input, Badge, Tooltip) use Tailwind correctly. Everything custom doesn't. This creates a two-tier styling system where Tailwind's JIT purging, responsive utilities, and design token enforcement don't apply to the bulk of the component tree. It also limits what AI tooling can reliably reason about — one of the explicit goals of standardizing on Tailwind.

This is understandable as a mockup shortcut — inline styles are fast to write when iterating on layout — but the deviation needs to be resolved before the component tree goes to production.

07

Orphaned Code

Low
Sidebar component

components/tortelligence/sidebar.tsx is 133 lines of placeholder nav and account area. It is never imported or rendered by AppShell or any other component. Either it belongs to an earlier design direction or it was superseded by the top-bar surface switcher. Should be deleted or wired up.

Original Firestore data path

lib/firestore.ts, lib/items.ts, and the getItems() function still exist from proto-next's original Firestore-backed home page. They are not called by anything in the Tortelligence UI. When the real data layer is built, these may become the foundation — but right now they're disconnected and should be treated as such.

08

What's Working Well

For the Record
✓ Store architecture is sound. The state/actions split, feature hook pattern, and persistence config are all done correctly. The foundation is right — the gaps are in what connects to it, not in the stores themselves.
✓ No local business state in components. Filters, selection, and surface preference all live where TS-00X says they should — in stores, accessed via hooks.
✓ Surface persistence works correctly. localStorage key, version, and partialize are all configured. Verified that reloading the page restores the active surface.
✓ shadcn/ui primitives used for interactive elements. Input and Tooltip are wired to external state, not self-contained — consistent with the standard's externally controlled primitives requirement.
✓ ClientView and SummaryFeed share one hook correctly. useOpportunityView accepts an optional module pin — SummaryFeed passes none, ClientView passes "pharma" — without duplicating filter logic.
