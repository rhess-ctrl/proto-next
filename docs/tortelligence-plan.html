What the mock contains

Two surfaces

S1 · Internal — Summary + Feed
Module health tiles (4 modules) at the top, then a filterable, ranked opportunity feed below. Clicking a module tile filters the feed.

S2 · Client — Single-module view
Firm-facing, scoped to one module (Pharma). Branded masthead, description copy, feature-density opportunity cards.

Design system

Full set of CSS custom properties: navy surface scale (abyss → panel → raised → well → trough), cyan primary, gold secondary, green/rose score semantics.

Three fonts: Oswald (display/headings), Open Sans (body), DM Mono (labels, metrics). 4px spacing grid. Restrained radius scale.

04

Zustand stores

opportunity-store.ts
type OpportunityState = {
  expandedId:    string | null
  activeModule:  string | null  // "all" or module key
  query:         string
}
type OpportunityActions = {
  setExpanded:   (id: string | null) => void
  setModule:     (m: string | null) => void
  setQuery:      (q: string) => void
}

Note: opportunities[] is not in this store — it's static seed data passed from the server layer or imported from a lib. The store owns only interactive UI state.

surface-store.ts
type SurfaceState = {
  active: "internal" | "client"
}
type SurfaceActions = {
  setSurface: (s: SurfaceState["active"]) => void
}

Uses Zustand persist middleware to survive page reloads (localStorage key "tortelligence.surface", version 1). This is the durable preference case from the TS standard.

Feature hooks (per TS-00X §Interaction Model)

Components don't import stores directly — they use thin feature hooks that select minimal state and expose typed callbacks:

// hooks/use-opportunity-view.ts
export function useOpportunityView() {
  const { expandedId, activeModule, query } = useOpportunityStore(s => s.state)
  const { setExpanded, setModule, setQuery }  = useOpportunityStore(s => s.actions)
  const filtered = useMemo(() => /* filter + sort */, [activeModule, query])
  return { filtered, expandedId, activeModule, query, setExpanded, setModule, setQuery }
}

07

Key decisions + tradeoffs

DecisionChosen approachWhy
InfoDot tooltip shadcn Tooltip (Radix Portal) The mock uses a custom ReactDOM.createPortal + viewport-clamp approach. shadcn Tooltip (Radix) handles portal + positioning out of the box with better a11y and no custom math.
Fonts next/font/google The mock bundles woff2 as base64 blobs. Next.js font optimization via next/font/google gives self-hosting, preloading, and zero layout shift — no manual woff2 management needed.
Design tokens CSS custom properties + Tailwind v4 @theme Tailwind v4 reads CSS custom properties natively. Defining tokens once in globals.css gives both CSS var access (for inline styles in complex components) and Tailwind utility class access.
Opportunity data Static seed in lib/opportunities.ts Mock uses window globals. In Next.js, a lib module is cleaner — server components can import it directly, client components via props or hooks. No Firestore needed for this data yet.
Server vs client Page = server; interactive components = 'use client' Page components (dashboard/page.tsx) can be server components that import static data and pass it down. Store-connected components (OppCard, TopBar, SummaryFeed) need 'use client'. This keeps RSC benefits at the route boundary.
Sparkline chart Custom SVG — no library The spark is a simple polyline over a series array — 25 lines of SVG. Bringing in recharts or a chart library for this would be overkill and adds bundle weight.
